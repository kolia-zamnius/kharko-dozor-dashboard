---
title: Display-name overrides
---

Whenever the dashboard renders a user (Replays list, Users list,
user detail header), it resolves what to show via a deterministic
4-step chain. The first step that yields a non-empty string wins.

## The chain (priority order)

1. **`customName`** (per user) ‚Äî explicit override typed by an
   admin. Wins over everything.
2. **`displayNameTraitKey`** (per user) ‚Äî read `traits[<this key>]`
   for this specific user. E.g. set `"email"` on a user ‚Üí dashboard
   shows their `traits.email`.
3. **`defaultDisplayNameTraitKey`** (per project) ‚Äî same idea, but
   applied as a default to every user in the project. Set to
   `"email"` once ‚Üí every user in the project with an `email` trait
   shows it.
4. **Fallback to `externalId`** ‚Äî opaque, but always available.

Trait values that aren't strings are coerced through `String(value)`
before display, so `plan: 42` with key `"plan"` renders as `"42"`
rather than silently falling through. Empty strings (post-trim) are
treated as absent ‚Äî the chain falls through.

The chain runs server-side as `resolveDisplayName(...)` and the
resolved value ships pre-baked in `userDisplayName` on every list
endpoint. Client code doesn't walk traits per row ‚Äî the work is done
once on the server.

## Where to set them

All three knobs live in **one modal** opened from the user detail
header. Click the pencil-edit icon next to the display name (visible
only to OWNER / ADMIN ‚Äî viewers don't see the trigger). Inside, three
stacked sections ‚Äî each with its own input + Save / Reset:

| Section                   | Field set                            | Scope                      |
| ------------------------- | ------------------------------------ | -------------------------- |
| **Custom name**           | `TrackedUser.customName`             | This user only             |
| **Trait key (this user)** | `TrackedUser.displayNameTraitKey`    | This user only             |
| **Project default**       | `Project.defaultDisplayNameTraitKey` | Every user in this project |

`null` clears any field ‚Äî chain falls through to the next step. A
non-empty string sets the override.

Both endpoints (`PATCH /api/tracked-users/[id]/display-name`,
`PATCH /api/projects/[id]/display-name-trait-key`) require ADMIN+ on
the owning org, double-validated server-side.

## Worked examples

### Common case ‚Äî set the project default once

Project `Production` has `defaultDisplayNameTraitKey = "email"`.
Three users in this project:

| User | Traits                                     |
| ---- | ------------------------------------------ |
| A    | `{ email: "alice@acme.com", plan: "pro" }` |
| B    | `{ name: "Bob", plan: "free" }`            |
| C    | `{ plan: "free" }` (no email, no name)     |

Resolved display names:

- A ‚Üí `"alice@acme.com"` ‚Äî step 3 hits (project default `"email"` matches A's traits)
- B ‚Üí `externalId of B` ‚Äî step 3 misses (no `email` trait on B); step 4 fallback
- C ‚Üí `externalId of C` ‚Äî same as B

### Per-user override

Same project, but for User B specifically you set
`displayNameTraitKey = "name"` on their detail page:

- A ‚Üí `"alice@acme.com"` (step 3)
- B ‚Üí `"Bob"` (step 2 wins ‚Äî per-user trait key beats project default)
- C ‚Üí `externalId of C` (fallback)

### Custom name wins everything

For User C, you type `"VIP customer (manual)"` into the **Custom
name** field:

- A ‚Üí `"alice@acme.com"` (step 3)
- B ‚Üí `"Bob"` (step 2)
- C ‚Üí `"VIP customer (manual)"` (step 1 wins over fallback)

## Anonymous sessions

Sessions without a `trackedUserId` (no `dozor.identify()` called) ship
with `userDisplayName: null` and the dashboard renders them as the
empty dash (`‚Äî`) in the User column. They don't appear at all in the
Users list ‚Äî only Replays.
